# Proposal: A Rust Port of the Spaghetti Detector

**Status**: Phase 5's local prep complete (2026-07-17) — see §10. The port itself has been done since Phase 4: all 36 rules, `674/674 issues identical` to Python, reproducible output, `--plan` byte-identical. **Decisions (2026-07-17)**: Python is spec of record (§5); new sibling repo `spaghetti-rs` (§9); standalone binary only for now, no wheel (§8); no specific benchmark target driving this — Phase 0 proves correctness, not speed. Full rationale in §11.

## 1. Summary

Build a Rust implementation of `spaghetti` alongside the existing Python one, not instead of it. Both stay maintained. The Rust build targets speed and zero-runtime-dependency distribution (a single static binary, no `uv`/Python install required to run it in someone else's CI); the Python build stays the reference implementation, the one this workspace's own `AGENTS.md`/`CLAUDE.md` runbooks already wire into `uv run spaghetti`, and the one that's cheapest to extend with a new rule.

The core risk isn't the port itself — every rule here is a pure `ast` tree query with no external I/O or LLM involvement, which is about as portable as static analysis gets. The core risk is **two implementations of the same 36 rules drifting apart** over time until "spaghetti says X" stops having a single meaning. Section 5 is the load-bearing part of this proposal; everything else is standard porting work.

## 2. Why

- **Speed at scale.** This workspace's own packages parse in well under a second, but the tool is explicitly designed to be pointed at *other* codebases (`--config`/`--package`) — arbitrary external repos of unknown size. `ast.parse` + `ast.walk` per file, times 36 checks, is fine at hundreds of files and starts to hurt at tens of thousands. A native implementation removes the interpreter overhead entirely.
- **Zero-dependency distribution.** Right now, running `spaghetti` anywhere requires a Python 3.13 environment and `pip install spaghetti-detector` (plus `boti` as a runtime dependency, pulled in only for the lightweight `Agent` base class — see §4.3). A static Rust binary can be dropped into any CI image, any pre-commit hook, any language's toolchain, with no interpreter at all.
- **A pure-`ast` tool is an unusually good porting candidate.** No network calls, no LLM calls (see §4.3), no filesystem writes beyond reading `.py` files and an optional YAML config. Every check function has the identical shape: `(tree: ast.Module, filepath, pkg) -> list[Issue]`. This is about as close to "just reimplement the pattern-match" as static analysis gets.

## 3. Non-goals

- **Not deprecating the Python version.** It remains the canonical spec — see §5.
- **Not chasing 100% rule parity on day one.** Ship a subset first (§6), grow coverage; a Rust build that covers 20 of 36 rules correctly is more useful than one that's stalled trying to cover all 36 before its first release.
- **Not porting `--plan` output formatting pixel-for-pixel on day one** — the remediation-plan feature (scoring.py `build_remediation_plan`/`plan_report`) is table-stakes valuable but can land in a later phase once the core scan/report/JSON path is solid.

Total: ~3,000 lines of implementation, ~1,700 lines of tests, 36 rules, 152 test cases.

### 4.1 The 36 rules aren't all equally hard

- **30 single-function AST checks** (`long-function`, `high-complexity`, `god-class`, `mutable-default`, `bare-except`, `magic-number`, etc.) — direct translations. Each is 15–60 lines of tree-walking against a handful of node types.
- **2 text checks** (`long-file`, `todo-marker`) — regex over lines, no AST needed at all.
- **3 cross-file checks** are the real work:
  - `import-cycle`: builds an intra-package import graph from `ast.ImportFrom`/`ast.Import` nodes and DFS-detects real cycles (not just a heuristic) — port is mechanical once the AST layer exists.
  - `duplicate-function-body`: groups functions by `ast.dump()`-equivalent structural identity. Rust's AST crates don't ship an "unparse/dump to canonical string" helper for free — needs a small custom structural-hash visitor (§7.2).
  - `sync-async-duplication`: needs `ast.unparse()` twice and a `difflib.SequenceMatcher.ratio()`-equivalent text-similarity score, compared against a configurable `--twin-similarity` threshold. This is the single highest-risk rule for behavioral parity (§7.3) — [CLAUDE.md](../CLAUDE.md) itself gates on this exact rule ("Sync-Async Twin Similarity: Must remain under 60% similarity"), so getting the ratio *wrong* would make the Rust build a worse gate than no gate at all.
  - `orphan-interface` (single-implementation ABC/`Protocol` detection): implemented and wired into `PACKAGE_CHECKS` in `checks/__init__.py`, but not in the README's documented rule catalog or the "36 rules" count — worth resolving on the Python side (adopt and document it, or remove it) before deciding whether to port it, rather than porting an ambiguously-scoped rule.

### 4.2 Everything else is genuinely portable

`models.py` (Issue/ScanResult dataclasses), `suppression.py` (two-line regex lookback), and `scoring.py` (weighted-sum arithmetic + threshold bands) have no Python-specific behavior to lose in translation — they're just data shapes and arithmetic.

### 4.3 What does *not* need porting: `boti.core.Agent`

`detector.py`'s `SpaghettiReviewAgent(Agent)` and `review_packages_concurrently()` look, from the name, like they might involve an LLM — they don't. `boti.core.Agent` here is a lightweight async-context-manager/lifecycle base class; the "review" is just `scan_package()` dispatched to a `ProcessPoolExecutor` worker, because **Python needs multiprocessing to get real parallelism past the GIL**. Rust has no GIL. This entire orchestration layer — `Agent` subclass, `ProcessPoolExecutor`, `asyncio.gather`, the `_init_worker` global-registry dance — collapses into a `rayon` `par_iter()` over packages (or even over files within a package, if per-package granularity proves too coarse). This is a real simplification the port gets for free, not a compatibility gap to paper over.

## 5. Keeping two implementations honest: a shared conformance suite

This is the part that actually determines whether this project is a good idea. Two hand-maintained rule sets for the same 36 rules **will** drift — a threshold tweak in one, a new edge case fixed in one, a new rule added to one — unless something mechanical stops it.

**Proposal: a golden fixture corpus, checked into a new shared location (a `spaghetti-conformance/` directory, or a third tiny repo `spaghetti-fixtures` both `spaghetti` and the new Rust repo pull in as a git submodule/subtree), consisting of:**

```
fixtures/
  long-function/
    case_basic.py            # input snippet
    case_basic.expected.json  # {rule, line, severity, message} — exact expected Issues
  sync-async-duplication/
    case_twin_pair.py
    case_twin_pair.expected.json
  ...
```

- Each existing Python test in `tests/test_detector.py` that constructs a small source snippet and asserts on the resulting `Issue`s is a candidate to be *extracted* into this format (many already are structured this way internally — they just live as Python string literals plus assertions rather than standalone files).
- **Both** `spaghetti` (Python) and the new Rust binary run a conformance-runner in CI that feeds every fixture through the tool and diffs actual vs. `expected.json`. A PR that changes rule behavior in one language without updating (or without the counterpart failing gate on) the shared fixture fails CI in *both* repos.
- Message text is allowed to differ (Rust's wording doesn't have to byte-match Python's) — but `rule`, `line`, and `severity` must match exactly. This keeps the suite from becoming a string-matching tax on normal wording improvements while still catching real behavioral drift.
- Thresholds (`MAX_FUNCTION_LINES = 50`, `COMPLEXITY_THRESHOLD = 10`, severity weights, fix-effort weights, etc. — currently hardcoded twice in `config.py`/`scoring.py`) should move into a single shared `rules.toml` that both implementations load at build/runtime, rather than staying as two independently-maintained Python/Rust constant tables. This turns "did someone forget to update the other language's threshold" from a code-review question into a non-issue.

**Decision this proposal recommends, open for pushback:** Python stays the **spec of record** — when the two disagree, Python's behavior is correct by definition and the Rust side has a bug, not the other way around. This matches "original version is also maintained" in the request, and it means new rules get designed/prototyped in Python first (cheap iteration) and ported to Rust once stable, rather than the reverse.

This isn't a novel design — it's how every widely-used spec with more than one language implementation already handles this problem, and the prior art tells us what matters:

- **CommonMark** ships `spec.json` — `{markdown, html}` pairs every implementation, in any language, is judged against. The spec *is* the executable test suite, not prose plus good intentions.
- **test262** (ECMAScript) and **WPT** (Web Platform Tests) are the shared suites every JS engine / browser engine runs in CI; a regression is a build failure across engines, not a bug report discovered months later by a confused user.
- **JSON-Schema-Test-Suite** plays the same role across dozens of independent validator implementations.
- **CPython as reference implementation** — PyPy/Jython don't negotiate with CPython when they disagree; CPython is correct by definition, with rare, explicitly documented exceptions. This is exactly the "Python is spec of record" framing above.

Two refinements this prior art argues for, on top of the mechanics already described:

- **Version the conformance suite itself, and have each implementation declare which version it passes** (a `CONFORMANCE.md` or a pinned version in each repo's CI config), rather than assuming perpetual lockstep. This is how Kubernetes conformance works — a distribution claims "conformant with v1.30," not "always in sync with upstream." It gives an honest, visible answer to "is the Rust build currently behind" instead of an unstated assumption that it never is. Concretely: tag the fixture corpus (`spaghetti-fixtures` repo or directory) with its own version; each of `spaghetti` and the Rust port's CI configs pin the fixture version they run against and bump it deliberately, the same way a dependency bump is a deliberate, reviewed step rather than automatic.
- **New rules get proposed as fixtures first, before either implementation writes code** — the same way a test262 proposal precedes engine implementation. A new rule starts life as an `expected.json` in the shared corpus with no implementation satisfying it yet; each repo's conformance runner treats "fixture exists, no implementation" as an explicit, visible, trackable **pending/skip** state (not a silent gap) until that repo's rule lands. This directly closes the "coverage drift" scenario from §5's opening paragraph — a 37th rule added only to Python no longer silently has no Rust equivalent; it shows up as a tracked, named gap in the Rust repo's own CI output until someone closes it.

## 7. Parity risk register

Ranked by how much custom work each needs beyond "translate the loop":

### 7.1 Parser crate stability — **spiked 2026-07-17, worse than expected, resolved by switching crates**
The concern as originally written ("no semver-stability promise") undersold it. `ruff_python_parser`/`ruff_python_ast` 0.0.5 don't just lack a stability guarantee — they're **broken as published**: `ruff_python_parser`'s own `Cargo.toml` unconditionally requires `ruff_python_ast`'s `get-size` feature (not optional, not something a downstream `Cargo.toml` can disable away), which pulls in `get-size2` 0.10.x's own `compact_str` 0.10.x dependency, while `ruff_python_parser` itself directly depends on `compact_str` 0.9.x — two SemVer-incompatible versions of the same crate produce two distinct `CompactString` types with the same name, and the `#[derive(GetSize)]` on `ruff_python_ast`'s `Name`/`ExprStringLiteral` types can't be satisfied for either. Crates.io shows only five ever-published versions (`0.0.0`–`0.0.5`), consistent with these being genuinely internal Ruff build artifacts, not a maintained public dependency — Astral doesn't publish them in lockstep with Ruff releases at all. **Resolution: switched to `rustpython-parser` 0.4.0** (the fallback this section already named), which added cleanly with no dependency conflicts and parsed all 111 real `.py` files in this workspace (boti, boti-data, boti-dask, spaghetti) with zero failures. Trade-off accepted: `rustpython-parser` may lag on the newest Python syntax versus whatever Ruff internally supports — not yet stress-tested against anything past what this workspace's own `py313`-target code happens to use.

### 7.2 `duplicate-function-body`'s structural equality — **resolved 2026-07-17, built one batch early**
Python's `_dump_stmts()` (in `ast_helpers.py`) leans on `ast.dump()`, which serializes a node's structure *and* identifiers while dropping position info (`lineno`/`col_offset`) — so two functions with identical logic but different variable names are correctly treated as *not* duplicates, while two functions differing only in source formatting/whitespace *are* correctly treated as duplicates. Turned out to need this primitive sooner than expected: Phase 2 batch 3's `duplicate-branch` rule (per-file, comparing if/else bodies) has the exact same requirement. Built `ast_helpers::dump_stmts()` — rather than hand-writing a field-by-field visitor across ~80 AST node variants, it uses `rustpython-ast`'s derived `Debug` output (which already recursively includes every field, including identifiers) and strips the embedded `range: N..M` byte-span fragments with a regex. Verified directly, not assumed: two blocks with different identifiers compare unequal; two textually-identical blocks at different source positions compare equal. Ready for Phase 3's `duplicate-function-body` to reuse as-is — no further spike needed.

### 7.3 `sync-async-duplication`'s similarity ratio — **spiked 2026-07-17, confirmed real divergence, `similar` crate rejected**
Ran both algorithms over 3 real sync/async twin pairs extracted from `boti-data` (`BaseDataCube.load`/`aload`, `BaseArtifact.before_load`/`abefore_load`, `after_load`/`aafter_load`), comparing Python's `difflib.SequenceMatcher(None, a, b).ratio()` against `similar::TextDiff` under all three of its algorithms (Myers, Patience, LCS) on the identical `ast.unparse()`-produced text:

| pair | python (`difflib`) | rust `similar` (all 3 algorithms agree) |
|---|---|---|
| `load`/`aload` | 0.6897 | 0.8331 |
| `before_load`/`abefore_load` | 0.9710 | 0.9710 |
| `after_load`/`aafter_load` | 0.9710 | 0.9710 |

Two of three pairs match exactly; the third — the pair with genuine control-flow restructuring (the async version adds a nested `if`/`elif`/`else` where the sync version uses a flatter `if ... and ...()` / `elif not ...()`), not just a mechanical `await`/`a`-prefix insertion — diverges by 0.14, regardless of which of `similar`'s three algorithms is used. This isn't a tuning/config problem: Myers/Patience/LCS are all shortest-edit-script-family algorithms that are effectively LCS-equivalent for alignment purposes, while Python's `SequenceMatcher` implements Ratcliff/Obershelp — a *different* algorithm (recursively finds the single longest contiguous matching block, then recurses on the remainders left and right of it), which is well-documented in the diff literature to not always find the same total match length as an LCS-family approach, especially once there's more than one equally-good longest block or the optimal global alignment requires passing over a locally-shorter match. Confirms §4.1's flagged risk with real data, not a hypothetical.

**Decision: reject the `similar` crate for this rule.** Implement Ratcliff/Obershelp directly in Rust instead (per this section's original fallback) — it's a well-specified, recursive, not-large algorithm, and it's the only way to guarantee `--twin-similarity 0.6` means the same thing Python's does. This is now committed Phase 3 scope, not a spike question.

**Resolved 2026-07-17**: implemented directly (`src/similarity.rs`), including `autojunk` — initially assumed skippable (real function bodies are usually well under the 200-character activation threshold), until Phase 0's own `load`/`aload` ground-truth pair (an unparsed `aload` body of 414 characters) produced a real 0.79-vs-0.69 mismatch that traced straight back to it. With autojunk implemented, all three of Phase 0's ground-truth ratios match exactly (0.6897, 0.9710, 0.9710).

A second, genuinely obscure discovery during Phase 3's real-code conformance run: `difflib.SequenceMatcher(None, a, b).ratio()` is **not symmetric** once autojunk is active — swapping which string is `a` and which is `b` changes the ratio, because "popular" (junk-like) characters are identified from `b` only. Confirmed directly by monkey-patching `SequenceMatcher.ratio` in the real Python tool: for `boti-data`'s `ChunkedLoadExecutor.load`/`aload`, the two call orders returned 0.4654 and 0.6192 for the *same pair*. Python's own dedup only marks a pair "reported" on success (`if ratio >= min_ratio: reported.add(...)`), so a pair whose forward direction falls short gets silently retried in the reverse direction on a later loop iteration — and if the reverse crosses the threshold, it fires, using that reverse ratio. This is a real, load-bearing quirk of the original tool, not a bug to smooth over: the Rust port's dedup (in `checks/package_level.rs`) replicates the identical "only mark reported on success" structure so the same retry-in-reverse behavior happens, and `similarity::sequence_matcher_ratio` is genuinely asymmetric the same way (`bpopular` computed from the second argument only) — verified directly against this exact pair, not assumed.

### 7.4 Workspace-root discovery semantics
`_find_workspace_root()` walks upward from the scanned path looking for a `pyproject.toml` with a `[tool.uv.workspace]` table, tolerating unreadable/malformed files along the way rather than crashing (this matters because `spaghetti` itself needs to import cleanly when pip-installed standalone into a repo with no such ancestor — its own CI checkout is exactly this case). The Rust port needs the identical "keep walking, don't error" tolerance, not a fail-fast walk.

### 7.5 Encoding tolerance
Python's `read_text(encoding="utf-8", errors="replace")` never throws on malformed UTF-8 in a scanned file — it substitutes replacement characters and keeps going. Whatever Rust file-reading path is used needs the same non-fatal-by-default behavior; a Rust `String::from_utf8` that just propagates an error on any bad file would make the tool less robust than the Python version it's supposed to complement, not just differently robust.

### 7.6 `rustpython-ast`'s generated `Visitor` silently skips several "support struct" internals — a whole family of the same bug, found across batches 1-2, fixed
Not a risk anticipated going in — found via conformance diffs against this workspace's real code, one instance at a time:

1. **Comprehensions** (batch 1): Python's `high-complexity` fired on two real functions (`boti-data/src/boti_data/gateway/_series_filters.py:39`, `boti-data/src/boti_data/pipelines/sinks.py:40`) that the Rust port missed — both had a comprehension whose `if` filter contained a multi-value `and` (`BoolOp`).
2. **Function arguments/defaults** (batch 2): `def f(x=[i for i in range(10) if i > 5 and i < 8]):` — Python's checks see the magic numbers and the `and` inside that default value; the Rust port saw none of it.
3. **Call keywords** (batch 2): `warnings.warn(msg, RuntimeWarning, stacklevel=3)` (`boti/src/boti/core/logger.py:217`) — Python's `magic-number` sees the `3` in `stacklevel=3`; also explained a `unused-import` false positive, since `started=perf_counter()` as a keyword argument made the usage invisible to the scan.
4. **`with`-items** (found by inspection, applying the same lesson pre-emptively rather than waiting for a fourth mismatch): `with open(path, encoding="utf-8") as f:` — the context-manager expression itself was never visited either.

Root cause, same in all four cases: `rustpython_ast::Visitor`'s auto-generated `generic_visit_comprehension`/`generic_visit_arguments`/`generic_visit_arg`/`generic_visit_keyword`/`generic_visit_withitem` are all no-op stubs — confirmed by reading `rustpython-ast-0.4.0/src/gen/visitor.rs` directly. Every *node*-shaped `generic_visit_*` method properly descends into all its fields; every *support-struct*-shaped one (anything that isn't itself a `Stmt`/`Expr` variant) is stubbed out empty instead. The call sites still invoke `visit_arguments`/`visit_keyword`/`visit_withitem`/`visit_comprehension` correctly (e.g. `generic_visit_expr_call` does call `self.visit_keyword(kw)` for each keyword) — the default's just a dead end once you get there.

**Fix**: four reusable helpers in `ast_helpers.rs` — `walk_comprehension_children()`, `walk_arguments_children()`, `walk_keyword_children()`, `walk_withitem_children()` — each documented with which real mismatch found it. Any visitor needing full `ast.walk()`-equivalent traversal must override the corresponding `visit_*` method and call the helper; applied so far to every visitor that does full-expression scanning (`ComplexityVisitor`, `MagicNumberVisitor`, and `check_unused_imports`'s usage scanner). Verified: with all four fixes in place, the Rust port matches Python exactly — **470/470 issues identical** across boti/boti-data/boti-dask for all 20 rules ported through batch 2.

This is exactly the kind of gap the port proposal's approach is designed to surface — a real conformance diff against real code, not a hand-built fixture, caught every instance. Worth remembering for batch 3 and beyond: any *new* visitor doing full-expression traversal should assume this crate's support-struct defaults are broken until proven otherwise, rather than trusting them by default.

### 7.7 Output wasn't reproducible run-to-run — a class of bug the conformance diff structurally can't catch, found 2026-07-17, fixed
Every verification through Phases 0-3 compared Python's and Rust's output as *sets* of `(file, line, severity, rule)` tuples — the right check for "do the two implementations agree," but blind by construction to whether either implementation agrees *with itself* across repeated runs. Phase 4's own repeatability check (running the finished binary 5 times against unchanged input) found real nondeterminism: `--json` output differed between runs on the exact same code, with zero code changes in between.

Root cause: several package-level checks (`duplicate-function-body`'s grouping, `import-cycle`'s graph traversal, `sync-async-duplication`'s candidate iteration) used plain `HashMap`/`HashSet`, whose iteration order is randomized per-process in Rust (a deliberate DoS-hardening default, unlike Python's `dict`, which has preserved insertion order as a language guarantee since 3.7). Every place that iteration order could leak into output order — which duplicate-group gets reported, which cycle "wins" when several exist, which candidate name gets tried first — was a potential source of run-to-run difference, even though the underlying analysis was fully correct in `rule`/`line`/`severity` content.

**Fix**: sorted every such iteration point (module names before the cycle-detection DFS, duplicate-function groups by `(file, line)` before emitting, `twin_candidates` returns a sorted `Vec` instead of a `HashSet`) or switched to an order-preserving `IndexMap` where insertion order itself was semantically meaningful (already done for `sync-async-duplication`'s cross-file dedup — see §7.3). Verified: 5 consecutive runs of the release binary against the same real-code input now produce byte-identical `--json` output.

Worth carrying forward as a standing check, not a one-time fix: a conformance suite that diffs against another implementation is necessary but not sufficient — it needs a same-implementation repeatability check alongside it, because "matches Python" and "matches itself" are genuinely different properties, and nothing about passing the first implies the second in a language where the standard hash-map type doesn't preserve order.

## 8. Distribution & packaging

**Decided (§11): standalone binary only for now — no wheel.**

1. **Native binary** (the whole deliverable): prebuilt release binaries (macOS arm64/x64, Linux x64/arm64, Windows) via `cargo-dist`, attached to GitHub Releases; also published to crates.io so `cargo install spaghetti-detector-rs` works. This is the artifact that actually delivers on "no Python runtime required."
2. **`pip install`-compatible wheel** — explicitly out of scope for now, not a committed stretch goal. If demand shows up later (someone actually wants `pip install spaghetti-detector[native]` rather than a direct binary download or `cargo install`), revisit as a thin `maturin`-built wheel that bundles the prebuilt binary and shells out to it. Not worth building speculatively — see §11.3.

## 9. Repo placement

Recommend a **new sibling repo** (e.g. `spaghetti-rs`), not a subdirectory inside the existing `spaghetti` Python repo. This matches how this workspace already treats `boti`/`boti-data`/`boti-dask`/`tripwire`/`spaghetti` — five independent, single-toolchain repos, each with its own CI/publish pipeline — rather than introducing the first polyglot repo into that set. The shared conformance fixtures (§5) are the connective tissue between the two repos, not a shared codebase.

## 11. Decisions (resolved 2026-07-17)

All four were open questions as of the initial draft; each is now settled and drives the phasing above directly.

1. **Spec of record: Python.** When the two disagree, Python's behavior is correct by definition and the Rust side has a bug, not the other way around — matching "original version is also maintained" and the CPython/PyPy precedent from §5. New rules get designed/prototyped in Python first (cheap iteration) and ported to Rust once stable.
2. **Repo placement: new sibling repo, `spaghetti-rs`.** Matches the workspace's existing one-language-per-repo convention (`boti`/`boti-data`/`boti-dask`/`tripwire`/`spaghetti` are each standalone with their own CI/publish pipeline) rather than introducing the first polyglot repo into that set. The shared conformance fixtures (§5) are the connective tissue between the two repos, not a shared codebase.
3. **Packaging: standalone binary only, no wheel.** `cargo install`/prebuilt GitHub Release binaries (§8.1) is the full initial scope. The `maturin`-wrapped `pip install spaghetti-detector[native]` wheel (§8.2) is explicitly deferred, not a committed stretch goal — build it only if actual demand for a `pip`-native install path shows up, rather than speculatively.
4. **No specific performance bottleneck driving this — the native binary itself is the win.** There's no external codebase/CI scan-time pain point to benchmark against. This means Phase 0's spike is scoped to **prove correctness** (does `ruff_python_parser` parse this workspace's real files cleanly; does the `similar`-crate ratio track `difflib`'s closely enough) rather than to hit a speed target — no benchmarking harness is needed before Phase 1. Performance validation can happen naturally once the port is functionally complete, as a "how much faster did we actually get" writeup rather than a gate.

## 12. Remaining steps requiring explicit go-ahead

Everything through Phase 5's local prep is reversible — files on disk, in a git repo with no remote, nothing pushed or published. Three concrete actions remain, and each is a step up in how hard it is to undo, so each gets its own explicit confirmation rather than being bundled into "finish packaging":

1. **Create `github.com/lvalverdeb/spaghetti-rs`** (per §9/§11.2's decision) and push the local repo to it. Reversible in principle (a repo can be deleted) but visible to anyone the moment it's public, and not something to do silently.
2. **Set up the `CARGO_REGISTRY_TOKEN` secret** on that repo so the `cargo-dist`-generated release workflow can publish on tag-push. Requires the user's own crates.io token — not something this session has or should have.
3. **Publish `spaghetti-detector-rs` to crates.io** — either by pushing a version tag (letting the generated CI do it) or running `cargo publish` directly. This is the one genuinely irreversible action in the whole list: a published version can be *yanked* (hidden from new dependents) but never fully removed from the index.

None of these have happened. Recommend doing them in that order, each with its own confirmation, whenever the user is ready — not as a single bundled "ship it."
